HDR: High Dynamic Range

Tone mapping

Map the high dynamic range image to a normalized (0-1) range

  • High exposure: dark -> brighter
  • bright values too bright -> can’t see detail
  • Low exposure: Dark values too dark, bright less bright

Many games simulate this with a dynamic exposure:

  • Compute the luminance of every pixel
  • Scale down, averaging neighboring pixels
  • Repeat until we have a single exposure value
  • Blend with exposure value from previous frame
  • Use that exposure value for tone-mapping

float rayMarch(vec3 origin, vec3 dir)
{
	float distance = 0.0f;
	for(int i = 0; i < maxSteps; ++i)
	{
		vec3 p = origin + dir * distance;
		float d = GetDistance(p);
		distance += d;
		if (distance > maxDistance)
			discard;
		if (d < surfaceDistance)
			break;
	}
return distance;
}

Feature of a Recommender System

  • Personalised Recommendations → different recommendations for every user;

  • Maintain a user model or user profile, describes the users’s preferences;

  • Account for the context and additional information;

Problem Setting

User-Item Matrix

Goal: predict ratings for unseen items

Properties of the Ratings Matrix

Sparse matrix: a small subset of entries are specified

  • Ratings can be:
    • Continuous: level of like or dislike (rare);
    • Interval based ratings: 5 or 7-point scale, sometimes 10 points, e.g., 5-star ratings;
    • Ordinal ratings: categorical values, e.g., strongly disagree, disagree, neutral, agree, strongly agree;
    • Binary ratings: 2 options (forced choice), e.g., like or dislike;
    • Unary ratings: only positive preference (positive preference utility matrix), e.g., purchase

Implicit / Explicit Feedback

  • Explicit feedback: ratings explicitly provided by users, e.g., click on “like” button;
  • Implicit feedback, sometimes Positive-Unlabeled (PU):
    • Ratings are inferred from user actions;
    • Easier to obtain, transparent to users;
    • Almost no cost;
    • Harder to interpret and can be misleading: no information whether the user liked/disliked an item;

Long-tail Distribution

• Low frequency items:

    • Larger profit;
    • More difficult to recommend;

Collaborative Filtering Recommendation

• Idea: exploit past behaviour/opinions of an existing user community to predict items the current users will like in the future (collective intelligence);

• Input: ratings matrix (pure CF)

• Output:

A. Predicted rate for unobserved user-item;

B. List of top-k recommended items (or users);

User-based nearest neighbour recommendation:

• Similar users have similar ratings on the same item; •

Use ratings from neighbouring users;

Item-based nearest neighbour recommendation:

• Reverse the problem → similarity between items;

• Similar items are rated in a similar way by the same user;

• Use user’s own ratings on neighbouring items.

User-based Nearest Neighbour Recommendation

• Idea: given a target user:

  1. Identify users (peer users or nearest neighbours) that have similar preferences;
  2. For every unobserved item, make a prediction based on the preferences of peer users;

• Assumptions:

​ • Similar tastes in the past → similar tastes in the future;

​ • Preferences are stable and consistent over time.

Step 1: Similarity with Pearson Coefficient

• Simple approach: users with highest similarity score (Pearson coefficient);

• Problem: number of observed ratings can be very different depending on the item, e.g., popular items;

Step 2: Mean-centered Rating Prediction

Solution: center ratings by their row-mean

Use a weighted average to predict the rating:

Better Neighbourhood Selection?

• Include all users?

​ • Negative impact on accuracy and computing time;

• Minimum threshold on user similarity:

​ • Only positive correlation (negative correlations → inversion → not much predictive value);

​ • High threshold → small sets of neighbours → no predictions (reduced coverage);

• Top-k closest neighbours:

​ • High value for k → additional noise;

​ • Empirically: good values for k are in the range of [20, 50]

Item-based Nearest Neighbour Recommendation

similarity between items instead of similarity between users

Works better with offline pre-processing

Step 1: Cosine Similarity

Cosine similarity: empirically better than Pearson for item-based recommendation

Adjusted cosine similarity: mean centred ratings ;

Step 2: Rating Prediction

Consider the top-k similar items on a user basis;

set of K closest items (adjusted cosine similarity) to the target item that were rated by user ;

Forward rendering

Describe how forward rendering works

Render the object with the first light + indirect lighting

if more than one light, render again in additive mode

Shader renders one light at a time

  • First pass
    • No blending
    • Writes to depth
    • Depth comparison by default GL_LESS
    • Compute indirect lighting
  • Additional passes:
    • Additive blending: GL_ONE, GL_ONE
    • No depth -> first pass did
    • Depth comparison GL_EQUAL\GL_LEQUAL
    • no indirect light

bad for many lights (light of small range)

Deferred rendering

Describe how deferred rendering works

Geometry pass

All the lights, reading from the g-buffers and computing
lighting

  • Render all objects
  • Render to textures (g-butters) geometry buffers, each texture stores specific data
  • Each object can have its own specific shader

Lighting pass

All the lights,

reading from the g-buffers and computing
lighting

  • Render all lights
  • For each light
    • Read information stored in g-buffers
    • Transform it to original form
      • reconstruct position from depth
      • Recover Z component of normal
      • undo any range transformations
  • Obtain view direction/ light direction and compute lighting equation
  • Lighting in view space: relative to the camera
    • Read normals from g-buffer
    • Light position or direction(passed as uniform) must be in view space
  • Render all the lights in additive passes

Advantages

Efficient for rendering many lights

decrease time complexity O(m * n) -> O(m) + O(n)

More data in g-buffers needed for special materials

  1. Shader complexity

handle different lighting models for different materials increases the complexity for ALL fragments

It can not handle transparent objects

Explanation: piece of information justifying why an item is recommended

Ablation is the simplest type of attribution

• Drop each feature (one at a time)

• Attribute the change in prediction to the ablated feature

Advantage

• Simple and intuitive to interpret Disadvantages

• Indirect and partial explainability

• Does not always explain output (output invariant to ablation)

• Improper accounting of interactive features

• Can be computationally expensive

Shapley value

Game where players collaborate to make profit

• Take all possible subsets of players (in all possible orderings)

• Compute marginal contribution of a player

​ • For each player entering a subset:

​ • profit that the player creates by entering the subset

​ • profit in the subset before the player entered

• At the end, compute the average marginal contribution of each player

• The higher the average marginal contribution, the higher the attribution

NNs may use only the simplest feature and remain invariant to more complex features, even when every complex feature has significantly greater predictive power than each simple feature

Gradient-based optimization may get stuck at local minima

• At the start of training, the gradient of the simplest feature has predominant role

• Drags the model towards the best decision boundary of the simplest feature

• The model may get stuck in there

Attribution (ablation, Shapley) for recommendation explainability:

feature importance to output

Good for:

• Generating an explanation for the user

• Debugging model predictions

• Analysing model robustness (irrelevant features getting attributed? relevant features not getting attributed?)

Challenges

Computationally costly for hundreds of features

Reductive formulation of “why this prediction”:

• Not explain how features interact

• Not explain the model logic (e.g. model that recommends a photo when there is an even number of black pixels in it, otherwise it does not recommend it)

• Training-agnostic (what training samples influenced the prediction)

Item based vs User based

• Item-based explanations are more intuitive & credible to users because users are familiar with the items they interacted with before

• User-based explanations are less convincing & trustworthy because the target user may know nothing about other “similar” users

Vertex shader

One input vertex -> at least position is provided as an attribute

One output vertex -> output position in clip coordinates to gl_Position / Can have any other output attributes

Geometry shader

Input： One primitive, a collection of vertex attributes from previous stage arrays

Output: zero or more output primitives, multiple vertices per invocation, not arrays

e.g One line primitive expands to 2 triangles

Fragment shader (in Fragment processing)

Input: attribute from last stage, usually vertex shader

Output: main output is a color

Most common format is 4 components RGBA within in (0.0 - 1.0)

If depth test enabled, additional depth output
